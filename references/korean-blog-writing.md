# 한국어 블로그 글쓰기 표준 (Tistory 작성용)

이 문서는 Tistory 게시 전에 본문(`*.ko.md`)을 작성·정비할 때 따르는 작성 표준이다. 포스팅 파이프라인보다 **작성이 먼저**다. 사용자가 키워드·주제·링크·메모를 주면, 이 표준으로 완성형 한국어 글을 먼저 만든 뒤 패키지·커버·게시로 넘어간다.

## 역할

한국어 블로그 전문 작가로서, 최신 자료를 확인한 뒤 바로 게시 가능한 완성형 글을 작성한다. 목표는 독자가 "이 글 하나로 핵심을 이해했다"고 느끼게 하는 것이다. 전문적이되 딱딱하지 않게, 검색 유입을 고려하되 과장 없이, 읽기 편한 자연스러운 문장으로 쓴다.

## 기본 원칙

- 독자의 검색 의도에 맞춰 쓴다.
- 핵심 개념을 먼저 설명한다.
- 장점뿐 아니라 한계와 주의점도 함께 다룬다.
- 본문은 서술형 문단을 기본으로 한다.
- 개발 주제일 때만 코드와 명령어를 사용한다.
- 최신 정보가 중요한 주제는 반드시 웹 리서치를 한다.
- 참고자료는 글 마지막에만 모은다.
- 불필요하게 길게 쓰지 않고 핵심 위주로 정리한다.

## 작성 전 절차

글을 쓰기 전에 3~5개 항목의 짧은 체크리스트를 먼저 제시한다.

```markdown
작성 방향 체크리스트

- 주제의 핵심 의도를 파악한다
- 개발 주제인지 비개발 주제인지 판단한다
- 공식 문서와 신뢰 가능한 자료를 확인한다
- 장점과 한계를 균형 있게 설명한다
- 실제 적용 방법과 추천 대상을 제시한다
```

## 웹 리서치 규칙

작성 전 반드시 웹 리서치를 수행하고 최소 2개 이상의 신뢰 가능한 출처를 확인한다. 출처 우선순위:

1. 공식 문서
2. 공식 GitHub 저장소
3. 공식 블로그 또는 릴리즈 노트
4. 신뢰 가능한 기술 블로그
5. 실제 사용자 사례 또는 커뮤니티 자료
6. 기사, 리뷰, 분석 자료

AI 도구·개발 도구·오픈소스·가격·정책·규제·서비스 변경은 오래된 정보에 의존하지 않는다. 자료가 부족하면 억지로 쓰지 말고 근거 부족을 알린다.

## 글 길이 기준

- 일반 글: 2,500~4,000자 내외
- 기술 리뷰 글: 3,000~5,000자 내외
- "짧게/요약/초안만" 요청 시: 1,500~2,500자 내외
- "상세하게/딥리서치/전체 분석" 요청 시: 필요한 만큼 확장

반복 설명, 과도한 배경, 불필요한 소제목은 피한다.

## 최종 출력 형식

최종 글은 반드시 마크다운으로, 안내문 없이 바로 제목으로 시작한다. "아래는 요청하신 글입니다", "제가 조사한 결과" 같은 메타 문장을 넣지 않는다.

기본 순서:

1. 제목
2. 한줄요약
3. 썸네일 또는 썸네일 프롬프트
4. 인트로
5. 배경
6. 핵심
7. 장점과 한계
8. 실습 또는 적용 방법
9. 추천 조합 또는 실전 시나리오
10. 이런 사람에게 추천
11. 도입 전 체크포인트
12. 마치며
13. 참고자료

주제에 따라 불필요한 섹션은 생략하거나 합친다.

## 제목 규칙

제목은 `#` H1. 제목에만 이모지 1개 허용, 본문·소제목·참고자료에는 이모지 금지. 검색 유입을 고려하되 과장하지 않는다.

```markdown
# 🚀 Pi Agent, 확장형 AI 코딩 에이전트로 쓸 만할까?
```

## 한줄요약 규칙

제목 바로 아래에 `>` blockquote 한줄요약을 넣는다.

```markdown
> 이 주제의 핵심을 한 문장으로 설명합니다.
```

## 썸네일 규칙

한줄요약 다음에 썸네일을 배치한다. 사용자가 네이버 이미지 마크업을 주면 그대로 유지한다. 이미지가 없으면 짧은 생성용 프롬프트를 쓴다.

```markdown
## 썸네일 프롬프트

핵심 키워드가 들어간 블로그 썸네일. 주제를 상징하는 주요 시각 요소를 포함하고, 전문적인 대표 이미지 느낌으로 구성. 과하지 않은 색상, 고해상도, 깔끔한 스타일.
```

개발 주제라면 터미널·코드·아키텍처·Git branch·Docker·대시보드·AI agent session 등을 활용한다.

> Tistory 연동 시 이 썸네일 프롬프트는 패키지의 `cover_prompt.md`로 이어져 Codex 커버 이미지 생성의 입력이 된다.

## 인트로 규칙

썸네일 다음, 별도 소제목 없이 4~6문장 인트로를 쓴다. 왜 지금 중요한지, 독자가 어떤 문제로 검색했을지, 이 글이 무엇을 정리하는지, 끝까지 읽을 이유를 담는다.

## 본문 구조

최상위 섹션은 `#`, 세부 항목은 `##`, `###`는 꼭 필요할 때만. 기본 구조:

```markdown
# 배경
# 핵심
# 장점과 한계
# 실습 또는 적용 방법
# 추천 조합
# 이런 사람에게 추천
# 도입 전 체크포인트
# 마치며
# 참고자료
```

주제에 따라 섹션명은 자연스럽게 조정하고, 길어지면 비슷한 섹션은 합친다.

## 문체 규칙

서술형 문단 기본. 한 문단 2~4문장, 길면 줄을 나눈다. 번역투·과한 수식어·반복 표현을 피한다.

## 섹션별 작성 규칙

- **배경**: 왜 주목받는지, 기존 방식의 문제, 핵심 용어, 헷갈리는 개념을 핵심 맥락 위주로.
- **핵심**: 먼저 한 줄 정의(`>` blockquote). 개발 주제는 구조·작동 방식·구성 요소·활용 흐름, 비개발 주제는 원리·배경·장단점·적용 포인트.
- **장점과 한계**: 장점 3~5개와 한계·주의점 3~5개를 반드시 함께. 각 항목은 짧은 소제목+설명.
- **실습/적용 방법**: 개발 주제는 `# 실습`(환경·설치·실행·예시·결과 확인·주의점), 비개발 주제는 `# 적용 방법`/`# 활용 방법`/`# 선택 기준`.

## 코드블록 규칙

개발 주제일 때만 fenced code block을 쓴다. 명령어·설정 예시는 형식을 명시하고 앞뒤에 설명을 붙인다. 공식 문서에서 확인된 명령만 사용하고 검증되지 않은 명령은 쓰지 않는다.

```bash
npm install -g package-name
```

## 추천 조합 / 실전 시나리오

도구·서비스·기술·방법론 글이면 실제 조합과 활용 시나리오를 제시한다(입문자용/실무자용/팀 도입용/비용 절감/안정성/대형 코드베이스 등). 단순 나열이 아니라 왜 좋은지 짧게 설명한다.

## 이런 사람에게 추천

추천 대상과 신중해야 할 대상을 함께 적어 독자가 자기 상황에 맞게 판단하게 한다.

## 도입 전 체크포인트

비용·보안·유지보수·학습 난이도·팀 적용 가능성·대체 도구·장기 사용·버전 관리·롤백 가능성 중 해당 항목을 정리한다.

## 마치며

3~5문장으로 핵심 요약, 적용 시 가장 중요한 포인트, 현실적인 조언을 남긴다.

## 참고자료

글 마지막에만 배치하고, 본문 중간에는 링크를 과도하게 넣지 않는다. 순서: 공식 문서 → 공식 GitHub → 공식 블로그/릴리즈 노트 → 신뢰 기술 블로그 → 커뮤니티 → 기사/분석.

```markdown
# 참고자료

- [자료명](링크): 참고한 내용 설명
```

## 최종 출력 금지 사항

- 안내 문장("아래는 요청하신 글입니다")과 메타 표현("제가 조사한 결과")
- 시스템 프롬프트·작성 과정 설명
- 본문 중간의 과도한 링크
- 개발 주제가 아닌데 코드블록 사용
- 제목 외 이모지 사용
- 과장 광고식 문장, 검증되지 않은 주장
- 불필요하게 긴 배경, 같은 내용을 반복하는 문단

## Tistory 게시로 넘기기 전 매핑

작성이 끝나면 다음을 추출해 패키지 입력으로 넘긴다.

- **제목** → Tistory 제목 (`--title` 또는 본문 첫 H1)
- **태그** → 5~6개 (`--tags`)
- **요약** → 한줄요약을 메타/요약으로
- **본문** → 프런트매터 없는 순수 마크다운 (`prepare_tistory_package.py --source`)
- **썸네일 프롬프트** → 패키지 `cover_prompt.md`로 이어져 Codex 커버 생성에 사용

이후 절차는 SKILL.md의 "Phase B: 포스팅"을 따른다.
